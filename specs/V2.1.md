## Problem Statement
The v2 site's email captures (the hero "What's your work email?" field and the closing-CTA form) are **visual-only** — they submit nowhere. The site looks finished but has **no working conversion**, which is the one job a pre-launch marketing site must do: capture interested design partners and notify the founders. This story wires those captures to a real Next.js server action that (a) validates input, (b) blocks spam, (c) emails the founders via Resend, and (d) sends the submitter an honest confirmation — with proper loading / success / error states. No database (email-only, per `CLAUDE.md` — no Supabase/Clerk).

## User Stories
- As a **design-partner ops leader**, I want to drop my work email (or the fuller form) and get a confirmation, so I know my interest registered and someone will reply.
- As a **founder (Nicolas)**, I want every request emailed to me with the submitter's context, so I can follow up within days.
- As **any visitor**, I want honest expectation-setting (who replies, when) and no broken/silent form.

## Technical Requirements
- **Files:**
  - `app/actions/request-access.ts` — `"use server"` action: parse → Zod validate → honeypot check → send via Resend → return typed `{ ok, errors? , message }`.
  - `lib/validation/request-access.ts` — the Zod schema (shared client/server).
  - `components/v2/request-access-form.tsx` — `"use client"` full form (closing CTA), `useActionState`, states, a11y.
  - `components/v2/email-capture.tsx` — `"use client"` hero quick-capture (email + submit), same action.
  - `emails/founder-notification.tsx` + `emails/submitter-confirmation.tsx` — React Email templates.
  - Wire these into `components/v2/hero.tsx` (replace the visual-only email input) and `components/v2/closing.tsx` (replace the visual-only form).
  - `src/scripts/verify-v2-1.ts` — checks below.
- **Env (`.env.example`, never commit real values):**
  - `RESEND_API_KEY` — from resend.com (Nicolas creates; never in the repo).
  - `REQUEST_ACCESS_TO` — founder inbox (e.g. `nibaranowski@gmail.com` for now).
  - `REQUEST_ACCESS_FROM` — a verified sender on `axonahq.com` (e.g. `access@axonahq.com`). **Resend requires domain verification** — until `axonahq.com` DNS is set up (deploy step), use Resend's `onboarding@resend.dev` sandbox sender so it works pre-DNS; swap to `access@axonahq.com` after DNS verification. Note this in `pre-launch-swap.md`.
- **Resend:** `npm i resend @react-email/components`. Initialize per-request (`new Resend(process.env.RESEND_API_KEY)`); guard if key missing (R9).
- **No DB:** leads exist only as the founder email. (If volume later warrants storage, that's a separate story.)
- **Security/privacy:** never put submitted PII in URLs/query strings; the action is POST via the form; honeypot + server validation only.

## UX Flow
```
Hero:  [ work email ] (Book a demo / Request access)
          └─ submit → requestAccess({email, source:hero})
                        ├─ invalid → inline error (retry)
                        ├─ honeypot filled → silent 200 (no send)
                        └─ ok → founder email + confirmation email → inline "Thanks — we'll be in touch…"

Closing form: Name · Work email · Company · Role · What are you building? · [x] consent → Request access
          └─ submit → requestAccess({...full, source:closing})
                        ├─ validation errors → field messages, focus first error
                        ├─ submitting → spinner, disabled
                        ├─ send fails / key unset → retryable error (graceful, no 500)
                        └─ ok → form replaced by success message; confirmation email sent
```

## Edge Cases
| Case | Handling |
|---|---|
| `RESEND_API_KEY` unset (pre-key / pre-deploy) | Validate + return friendly message (or log lead); no throw. Document. |
| `axonahq.com` not yet DNS-verified in Resend | Use `onboarding@resend.dev` sandbox `from` until verified; then swap. |
| Bot fills honeypot | Silent success, no email sent. |
| Duplicate / repeat submit | Accept; optionally debounce client-side. No dedupe DB. |
| Resend API error / network fail | Catch → retryable inline error; log server-side; never dead-end. |
| JS disabled | Server action still works via native form POST (progressive enhancement) — form submits and returns the result page/state. |
| Invalid email | Zod rejects server-side; client shows the message. |

## Common Mistakes to Avoid
- **Throwing when the key is unset.** Guard `RESEND_API_KEY`; the site must be deployable before the key exists — degrade gracefully (R9), never 500.
- **Client-only validation.** The server action re-validates with Zod; never trust the client.
- **`from` on an unverified domain.** Resend rejects unverified senders — use `onboarding@resend.dev` until `axonahq.com` is verified, then swap (track in pre-launch-swap).
- **PII in URLs.** POST only; never query strings.
- **Dead-end errors.** Every failure path is retryable + announced; show real loading/success/error states.
- **Honeypot via `display:none` only / no aria-hidden.** Hide it off-screen + `aria-hidden` so SR users don't hit it and bots still do.
- **Inventing fields/copy.** Use the `content.md` form fields + success copy verbatim.
